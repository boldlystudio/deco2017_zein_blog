# Designing BakeLab’s Data Structure for Feasible Development

As BakeLab became more defined as a product, I realised the project was no longer only about interface design. The app also needed a data structure capable of supporting structured bake analysis, critique workflows, and scalable interaction between users.

This stage focused on a core development question:

> What data structure best supports structured bake analysis and critique?

The challenge was balancing technical structure with realistic implementation scope. Instead of designing the “perfect” database system, I focused on creating a schema that was understandable, feasible, and aligned with the project’s minimum viable product (MVP).

## Why a relational database fits BakeLab

BakeLab contains highly structured information. Users create bake entries, other users leave critique, and each bake contains multiple process variables such as hydration, dough temperature, flour mix, and fermentation details.

Because these pieces of information are connected through clear relationships, a relational database made more sense than loosely structured storage.

For example:

- one user can create many bakes
- one bake can receive many comments
- one user can submit many critiques
- starter logs belong to individual users

These relationships are easier to manage with relational models because the database can maintain consistency between connected records.

This also supports future querying possibilities. For example, BakeLab could later allow users to:

- compare bakes with similar hydration levels
- filter critiques by feedback category
- analyse common fermentation problems
- inspect patterns across different flour combinations

A relational structure creates a stronger foundation for this kind of analysis.

## Why SQLite was the right decision

I selected SQLite because it matched the scope and learning goals of the project.

At first, I considered whether a larger database system would look more “professional.” However, after analysing the actual requirements of the app, SQLite became the more reasonable choice.

SQLite works well for BakeLab because it is:

- lightweight
- easy to debug
- simple to set up
- course-aligned
- well suited for local development

Since BakeLab is currently an MVP prototype rather than a production-scale platform, introducing more complex infrastructure would increase development difficulty without significantly improving the user experience.

This became an important design lesson: technical decisions should match the actual problem being solved, not the desire to appear technically advanced.

## Structuring the main tables

The database structure was intentionally designed around the core learning loop of the app:

**share bake → inspect process → receive critique → improve future bakes**

To support this flow, I focused on a small number of core tables.

### Users table

The users table stores account-level information and acts as the parent relationship for other content.

Example fields include:

- id
- username
- email
- created_at

This allows each bake and critique to be connected back to a user.

### Bakes table

The bakes table became the central structure of the system because almost every interaction revolves around bake analysis.

Example fields include:

- title
- hydration
- flour_mix
- starter_activity
- room_temp
- dough_temp
- fold_count
- bake_temp
- result
- question
- notes
- photo_url
- feedback_types
- created_at

This structure supports BakeLab’s goal of treating each bake like a diagnosis record rather than a simple social post.

### Comments and critique table

The comments table stores structured feedback from other bakers.

Instead of only allowing generic comments, the schema includes fields for:

- feedback_type
- confidence
- content

This supports more targeted critique and creates stronger alignment between the database structure and the UX goals of the project.

## Important trade-offs and simplifications


One of the most important parts of this stage was recognising where simplification was necessary.

Initially, I considered implementing a more complex many-to-many relationship for feedback categories. For example, each bake could connect to multiple feedback tags through a linking table.

Technically, this would create a cleaner relational structure.
