# Structuring Bake Feedback Without Overwhelming Users

BakeLab began with one key question: **how can a baking community encourage useful critique instead of chaotic comments?**

Many online food communities are good for inspiration, but not always good for learning. A baker might post a photo and receive comments like “looks great” or “try again next time.” These responses are friendly, but they do not explain what worked, what failed, or what could be improved. For a sourdough baker, that is a problem because small process choices can change the final result.

For this reason, I designed BakeLab less like a normal social feed and more like a structured feedback space. The aim is not only to share bakes, but to help users understand the relationship between process, result, and critique.

## Designing for useful critique

The main design decision was to guide feedback instead of leaving it completely open. A normal comment box gives users freedom, but it can also create low-quality responses. In BakeLab, the critique form asks users to choose a feedback category, such as crumb, oven spring, crust, shaping, fermentation, or scoring.

This makes the feedback more specific. Instead of writing a general opinion, the user is encouraged to focus on one part of the bake. I also included a confidence rating because not every baker has the same experience level. This matters because advice from a beginner should not be presented in the same way as advice from someone with stronger baking knowledge.

This decision supports trust. Users can still contribute, but the interface gives more context to their response.

## Information architecture decisions

BakeLab contains a lot of information: hydration, flour mix, starter activity, room temperature, dough temperature, fold count, bake temperature, result notes, questions, photos, and comments. If all of this information appears on one long page, the user can easily feel overwhelmed.

To reduce this, I organised the bake detail page into sections. The most important content appears first: the bake title, image, result, and the baker’s main question. More detailed process data is separated into tabbed sections.

This creates a clearer hierarchy:

1. **What happened?**  
   The user sees the result and the main issue.

2. **Why might it have happened?**  
   The user checks the process data.

3. **What feedback can I give?**  
   The user submits a guided critique.

4. **What can the baker learn next?**  
   The baker compares advice with their own notes.

This structure helps the page feel more like a diagnosis page than a post page.

## Reducing cognitive load

One challenge was deciding how much information to show at once. Sourdough baking is technical, but the interface should not punish users with a giant form or a crowded detail page.

I used progressive disclosure as a design approach. This means the interface shows the most important information first, then lets the user inspect more detail when needed. For example, a visitor does not need to see every variable immediately. They might first look at the photo, read the baker’s question, then open the process tab if they want to give deeper feedback.

This also improves the user flow because users are not forced to read everything before taking action.

## Why desktop-first matters

BakeLab was designed as a desktop-first web app because the main interaction is comparison-heavy. Users are not only scrolling through images. They are reading process data, checking patterns, writing critique, and comparing outcomes.

A mobile layout would be useful later, but desktop is more suitable for the first prototype because it gives enough space for:

- bake photos and notes
- process variables
- tabbed information
- critique forms
- comment history
- side-by-side comparison

This decision is connected to the purpose of the app. BakeLab is not only for quick reactions. It is for slower, more thoughtful learning.

## User flow reasoning

The main user flow is:

**Browse bake → inspect details → submit critique → compare results**

This flow keeps the experience focused. The user first scans the community feed to find a bake that interests them. Then they open the bake detail page to understand the result and process. After that, they submit structured critique based on a specific category. Finally, the baker can compare the critique with their own process notes and apply the learning to future bakes.

This flow also supports the minimum viable version of the app. Instead of trying to build every possible community feature, I focused on the core learning loop: sharing, inspecting, critiquing, and improving.

## Reflection

The biggest improvement in this stage was moving from “users can comment on bakes” to “users can give structured critique that supports learning.” That shift made the product direction clearer.

I also realised that information architecture is not just about organising content neatly. It affects the quality of user behaviour. If the page is messy, users may give shallow feedback. If the page is structured well, users are more likely to understand the bake and respond with care.

For the next stage, I need to test whether the critique categories are clear enough for users. I also need to check whether the tabbed layout helps users understand the bake or hides too much information. This will help me refine the prototype before final development.
