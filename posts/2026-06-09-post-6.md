## Introduction

As BakeLab moved from implementation into evaluation, the focus shifted from building features to understanding how well those features actually worked. Earlier blog posts explored planning, architecture, accessibility, and development decisions. This final reflection evaluates the completed application through performance audits, usability testing, accessibility reviews, and a reassessment of the original project requirements.

BakeLab was designed to help sourdough bakers diagnose problems, document their baking process, and receive structured feedback. Rather than functioning as a typical social platform, it focused on learning through critique and shared experience.

I established the GitHub repository and contributed to the application's architecture, database schema, controller logic, Community Feed, Bake Detail, New Bake pages, photo uploads, and accessibility improvements. My teammates Belle and Ahmad Zairin developed the Starter Log, My Bakes pages, weather API integration, and user testing activities. While testing evidence was shared across the team, this reflection focuses on my own observations and learning.

## Evaluating Performance

Lighthouse audits were conducted on the deployed application to evaluate performance and technical quality.

    Lighthouse audit results across BakeLab pages.

A key achievement was a Best Practices score of 100 across all pages, indicating that the application avoided common implementation issues and followed modern web standards.

HTMX also contributed positively to performance by updating only relevant page sections rather than reloading entire pages. Lighthouse reported zero Total Blocking Time and zero Cumulative Layout Shift across the audited pages, suggesting responsive interactions and stable layouts.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/bakelab-community-feed-lighthouse-testing-updated.png" alt="Community Feed Lighthouse audit">
  <p class="image-caption">
    Community Feed Lighthouse audit results after deployment to Render.
  </p>
</div>

The Community Feed recorded the lowest performance score at 89. After reviewing the audit results, the cause became clear. Unlike text-heavy pages such as Starter Log, the Community Feed loads multiple user-uploaded photographs simultaneously.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/bakelab-community-feed-accessibility-lighthouse-testing.png" alt="Community Feed Lighthouse opportunities">
  <p class="image-caption">
    Lighthouse identifying image delivery and accessibility as the main opportunities for improvement.
  </p>
</div>

Lighthouse estimated significant savings through image optimisation and reduced request latency. Because lazy loading was not implemented, all images load immediately when the page opens. This explains why the Community Feed performed slightly worse than the other pages.

SEO scores were also slightly lower because the application currently lacks meta descriptions. While this does not affect functionality, it reduces discoverability and highlights a feature that was overlooked during development.

Overall, the audits suggest that the application's technical architecture was appropriate for a prototype. However, the evaluation also revealed that decisions made for development speed, such as local image storage and the absence of lazy loading, created trade-offs that would become more noticeable as the application scales.

## Evaluating User Experience and Accessibility

Technical performance alone does not determine whether an application is successful. Users must also be able to understand the interface, complete tasks efficiently, and feel confident using the system.

To evaluate usability, five participants with varying baking experience completed four tasks:

- Log a bake
- Share a bake with the community
- Leave structured feedback
- Record a starter log

All participants successfully completed the tasks and located key features without assistance. This suggested that the application's navigation and information architecture effectively supported its primary workflows.

The structured feedback system received the most positive responses.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/usability-findings-and-insights.png" alt="Usability findings and insights">
  <p class="image-caption">

One participant explained that the confidence rating "made me feel more comfortable giving advice", while another described it as "the best part of the app". These comments suggest that the feature not only helped readers evaluate feedback quality but also reduced uncertainty around participation.

This finding was particularly interesting because the confidence rating was originally designed to help users judge the reliability of feedback. User testing revealed a second benefit: it encouraged participation by making contributors feel more comfortable sharing their opinions.

In contrast, the largest usability issue involved the Flour Mix field. Participants were unsure how to enter flour types, percentages, and abbreviations. Initially, this appeared to be a user interface problem, but testing revealed a deeper issue.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/schema-draft.png" alt="Schema showing flour_mix field">
  <p class="image-caption">
    Original schema showing flour_mix stored as a single text field.
  </p>
</div>

The database stores `flour_mix` as a free-text field. While this simplified implementation, it relied on users creating consistent formats themselves. Testing showed that this assumption was unrealistic. The issue demonstrated how database design decisions can directly affect usability outcomes.

Accessibility testing also informed several improvements, including skip navigation links, improved heading hierarchy, accessible radio button groups, ARIA labels, and colour contrast improvements.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/contrast-checker-after.png" alt="Contrast checker validation">
  <p class="image-caption">
    WebAIM contrast checker confirming WCAG AA compliant colour contrast.
  </p>
</div>

These changes contributed to accessibility scores between 94 and 95. However, Lighthouse still identified minor contrast issues on the deployed application, highlighting the need for ongoing accessibility testing rather than relying solely on local checks.

## Critical Reflection and Improvement Planning

The evaluation process helped identify not only what went wrong, but also why.

The first issue involved the Share and Unshare workflow. Although the interface appeared complete, parts of the backend functionality remained unfinished. This revealed that the team focused more on validating visual behaviour than testing complete end-to-end functionality.

The second issue was the Flour Mix field. Testing demonstrated that better labels or placeholder text would not solve the problem. The schema itself requires redesign. Future versions should use structured inputs such as flour selectors and percentage fields rather than relying on free-text entry.

The third issue involved accessibility. Despite resolving most contrast issues locally, some remained after deployment. This reinforced the importance of auditing production environments rather than relying entirely on local testing.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/future-improvements.png" alt="Future improvements roadmap">
  <p class="image-caption">

Based on the findings, my priorities for future development would be:

1. Replace the Flour Mix free-text field with structured inputs.
2. Complete and thoroughly test the Share and Unshare workflow.
3. Conduct a comprehensive accessibility audit on the deployed application.
4. Add contextual help for baking terminology.
5. Improve image loading through optimisation and lazy loading.

These recommendations focus on addressing the root causes identified during testing rather than simply adding new features.

## Retrospective Assessment of Functional Requirements

The final stage of reflection involved reassessing the original project requirements in light of what was ultimately built and evaluated.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/comparison-table.png" alt="Comparison between planned and delivered functionality">
  <p class="image-caption">
    Comparison between planned and delivered functionality.
  </p>
</div>

Most planned functionality was successfully implemented. However, several requirements emerged during development that were not anticipated in the original design, including flash messages, empty-state handling, owner-specific page behaviour, and additional accessibility requirements.

<div class="blog-image">
  <img src="/deco2017_zein_blog/assets/images/bakelab-ddd.png" alt="Original BakeLab Data Definition Document">
  <p class="image-caption">
    Original Data Definition Document showing the planned database structure.
  </p>
</div>

Looking back, most of the original requirements were realistic and appropriately prioritised because they focused on core baking workflows. However, the project underestimated the importance of accessibility requirements and overestimated the suitability of free-text inputs such as Flour Mix.

User testing revealed that some requirements needed additional structure, while others, such as flash messages and empty-state handling, were missing altogether. This demonstrated that requirements are not fixed decisions but assumptions that should be revisited throughout development.

The structured critique system was included in the original requirements and remained one of the project's core features throughout development. However, user testing revealed an outcome that was never explicitly anticipated during planning. Participants repeatedly reported feeling more confident contributing feedback because the confidence rating system reduced anxiety about giving incorrect advice.

While the feature was originally designed to help users evaluate the reliability of feedback, it also encouraged participation by making contributors feel more comfortable sharing their opinions. This demonstrated that the value of the feature extended beyond information quality and into user confidence and community engagement.

## Lessons Learned

BakeLab reinforced the importance of validating assumptions throughout development. Several issues identified during testing, particularly the Flour Mix field, originated from decisions that seemed reasonable during implementation but were never evaluated with users beforehand.

If I were to approach the project again, I would introduce usability testing much earlier in the process. Even a single session with a paper prototype would likely have revealed the Flour Mix issue before the database schema was finalised.

From a technical perspective, the project strengthened my understanding of TypeScript, Mojo.js, SQLite database design, accessibility auditing, Lighthouse evaluation, GitHub workflows, and full-stack development.

What I did not expect was that users responded most strongly to how the application made them feel less anxious about participating. While technical requirements can be planned in advance, confidence and trust often emerge through interaction design decisions.

This experience reinforced that successful web applications are not only functional systems but also social environments that influence how people contribute, learn, and engage with one another.
