As BakeLab developed into a more structured web application, evaluation and responsible design became increasingly important. Earlier stages focused mainly on defining the problem space and planning core functionality. However, as the project evolved, it became necessary to consider how users would navigate the critique workflow, whether the interface would remain accessible, and how user-generated content should be handled responsibly.

Although formal usability testing has not yet been conducted due to the assignment timeframe, this stage still involved evaluating anticipated usability issues, accessibility concerns, and technical limitations through iterative reflection and wireframe refinement.

# Evaluating Usability

One of the main usability goals for BakeLab is ensuring that users can navigate the critique workflow clearly and efficiently. Since the platform depends on users documenting baking processes and receiving structured feedback, confusion within the submission flow could reduce engagement and discourage critique participation.

Early wireframes revealed that the bake submission form risked becoming overwhelming because users were expected to enter hydration percentages, temperatures, flour types, notes, and questions simultaneously. This exposed a tension between collecting detailed bake data and maintaining readability.

To reduce cognitive overload, the form structure was reorganised into clearer sections with improved spacing and labels. The interface also prioritised readability over decorative interaction patterns because BakeLab’s main purpose is analysis and learning rather than social browsing.

If usability testing were conducted in a later stage, the evaluation would focus on:
- whether users understand critique categories
- where users hesitate during bake submission
- whether users can interpret bake data efficiently
- how clearly users understand visibility settings



# Accessibility and Responsibility

Accessibility became important because BakeLab contains large amounts of structured information and repeated form interaction. Several decisions therefore focused on improving readability and reducing interaction friction, including:
- semantic HTML structure
- readable typography
- keyboard-friendly navigation
- sufficient colour contrast
- clearly labelled form fields

Visible labels were prioritised instead of placeholder-only inputs because users need to compare and review detailed baking variables carefully. Consistent spacing and hierarchy also became important because overly dense layouts reduced readability during early wireframe exploration.

The project also considers responsibility in relation to user-generated content. Since users upload bake data and images, the planned system includes visibility settings that allow users to decide whether their bake appears publicly within the community feed.

# Reflection on Scope

One of the most significant changes during development involved reducing unnecessary complexity. Earlier ideas included AI-generated baking advice, advanced analytics, and more socially driven interaction systems. However, continued feasibility analysis showed that these features were beginning to weaken the clarity of the critique workflow itself.

As a result, the project became more focused on structured critique, readability, and maintainable interaction. Simplifying the architecture also improved implementation feasibility and reduced debugging complexity within the assignment timeframe.

This process reinforced an important lesson: effective design is not always about expanding functionality. In many cases, removing unnecessary systems creates a more understandable and achievable product.

# Conclusion

Although formal usability testing has not yet been completed, evaluation and responsible design still played an important role in shaping BakeLab. Reflection throughout development helped identify usability risks, accessibility concerns, and unnecessary complexity before implementation progressed further.

Most importantly, this stage demonstrated that responsible interaction design requires ongoing refinement and critical scope management rather than simply adding more features.
