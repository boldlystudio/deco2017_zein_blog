# Introduction

As BakeLab developed from an initial idea into a more structured web application, evaluation and responsible design became important parts of the project. Earlier stages mainly focused on defining the problem, planning features, and designing the overall workflow. However, as development progressed, it became necessary to consider how users would interact with the platform, whether the interface would remain accessible, and how user-generated content should be handled responsibly.

At this stage of the project, the prototype has not yet been fully implemented, which means formal user testing has not been conducted. Due to the assessment timeline and the blog deadline on 18 May, there was not enough time to complete development and organise usability testing sessions. Instead, this evaluation focuses on design decisions, anticipated usability concerns, accessibility planning, and reflections on how the project scope evolved during development.

# Evaluating Usability

One of the main usability goals for BakeLab is ensuring that users can navigate the critique workflow clearly and efficiently. The platform depends on users documenting baking processes and receiving structured feedback from others. If the submission process becomes confusing or overwhelming, users may avoid engaging with the system.

Early wireframes highlighted potential usability issues, particularly within the bake submission form. Users are expected to enter detailed information such as hydration percentages, flour types, temperatures, and notes. Displaying too many fields at once risked increasing cognitive overload and reducing readability.

To address this, the form structure was simplified into clearer sections with improved spacing and labels. The interface was also designed to prioritise readability over visual complexity. Since BakeLab focuses on learning and critique rather than social interaction, maintaining a clean and structured layout became more important than adding unnecessary visual effects.

Although user testing has not yet been completed, these design decisions were informed by common usability principles and iterative reflection throughout development.

# Accessibility and Responsibility

Accessibility was another important consideration because BakeLab contains large amounts of structured information and form interaction. Several accessibility-focused decisions influenced the interface design, including:

* semantic HTML structure
* readable font sizing
* sufficient colour contrast
* keyboard-friendly navigation
* clearly labelled form fields

Visible labels were prioritised instead of relying only on placeholders, improving clarity and supporting screen-reader accessibility. Consistent spacing and hierarchy were also important because the interface contains many technical baking variables that could otherwise become visually overwhelming.

The project also considers responsibility in relation to user-generated content. BakeLab allows users to upload bake information and images, meaning users should have control over what they choose to share publicly. To support this, the planned data structure includes visibility settings that allow users to decide whether a bake appears in the public feed.

# Reflection on Scope and Technical Decisions

As development continued, several earlier assumptions changed. Initially, the project included ideas such as AI-generated baking advice, advanced analytics, and more social interaction systems. However, further feasibility analysis showed that these features would introduce unnecessary complexity within the assignment timeframe.

As a result, the project became more focused on structured critique, readability, and maintainable architecture. Technical decisions such as using server-rendered interactions and avoiding overly complex frontend systems also helped keep the application manageable.

This process reinforced an important lesson: effective design is not always about adding more features. In many cases, simplifying the system creates a clearer and more achievable product.

# Conclusion

Although formal user testing has not yet been completed, evaluating usability, accessibility, and responsibility still played an important role in shaping BakeLab. The project evolved toward a more focused and maintainable design by prioritising readability, structured interaction, and realistic technical scope.

This stage of development highlighted that responsible design involves continuous reflection and refinement. Rather than attempting to implement every possible feature, the project became stronger by concentrating on the interactions that most directly support learning and constructive critique.
