One of the major technical decisions during the development of BakeLab was deciding how the application should handle interaction and page updates. At the beginning of the project, I considered using a modern frontend framework such as React because it is commonly used for interactive web applications. However, after reviewing the project requirements, development timeframe, and intended user experience, I decided that a simpler server-rendered architecture using Mojo.js and htmx would be more appropriate.

BakeLab is designed as a platform where users can document baking experiments, upload bake logs, and receive structured critique from other users. Most interactions involve reading information, submitting forms, and viewing feedback rather than highly dynamic real-time behaviour. Because of this, a large client-side framework would introduce unnecessary complexity without significantly improving the experience for users.

This decision became an important example of balancing technical ambition with project scope. Instead of selecting technologies based on popularity, I focused on choosing tools that aligned with the actual interaction requirements of the prototype.

# Choosing Mojo.js

BakeLab uses Mojo.js as the primary backend framework because it supports server-rendered interaction while remaining lightweight and manageable within the assignment scope. One advantage of Mojo.js is that routing, templates, and backend logic can remain closely connected. This creates a clearer development workflow and makes debugging easier during implementation.

For example, when a user submits a bake log, the server can process the form data, validate the inputs, store the information in SQLite, and return an updated page without requiring a separate frontend application. Since BakeLab relies heavily on structured forms and relational data, this workflow fits the project well.

Another reason for choosing Mojo.js was feasibility. The project already includes database modelling, CRUD operations, authentication, and interface design. Introducing a more complicated frontend stack would increase setup time and debugging overhead. By keeping the architecture simpler, I could focus more on solving the actual user problem rather than managing framework complexity.

# Using htmx for Interaction

Although BakeLab does not require a fully client-side application, it still benefits from smoother interaction patterns. To support this, I decided to use htmx alongside server-rendered templates.

htmx allows sections of a page to update dynamically without requiring a full page refresh. For example, critique comments or validation feedback can appear immediately after form submission while the rest of the interface remains unchanged. This creates a more responsive experience without introducing complicated frontend state management.

Another advantage of htmx is progressive enhancement. Since the server remains responsible for rendering content, the application can still function even if JavaScript fails. This improves reliability and keeps the architecture easier to understand during development.

Compared with a framework such as React, htmx also reduces frontend complexity. There is no need to manage component state, client-side routing, or API synchronisation. Instead, interaction logic remains closely tied to server-rendered HTML, which is more suitable for the scale of this prototype.

# Trade-Offs and Reflection

Choosing a simpler architecture also involved trade-offs. A server-rendered approach may not scale as effectively for highly interactive features such as live collaboration or real-time notifications. However, these features are outside the scope of the current BakeLab prototype.

For this project, the simpler architecture provides several practical advantages:

* easier debugging
* clearer data flow
* reduced frontend complexity
* faster development iteration
* stronger alignment with assignment scope

This decision also reinforced an important lesson about interaction design and development: technologies should support user goals rather than being selected purely because they are popular. BakeLab’s core interactions revolve around structured forms, critique systems, and readable content. These requirements do not justify the overhead of a large frontend framework.

# Conclusion

The technical architecture behind BakeLab was shaped by feasibility, usability, and project scope. Instead of building a highly complex frontend application, the project focuses on reliable server-rendered interaction using Mojo.js, SQLite, and htmx.

This approach supports the main goals of BakeLab while keeping development manageable within the semester timeframe. More importantly, the process highlighted how technical decisions are closely connected to interaction design decisions. Selecting the right tools is not about choosing the most advanced technology, but about choosing the technology that best supports the intended user experience.
