# Introduction

One of the main technical decisions during BakeLab’s development was deciding how the application should handle interaction and page updates. Initially, I considered using React because it is commonly associated with interactive web applications. However, after reviewing the project requirements, development timeframe, and user workflow, I concluded that a simpler server-rendered architecture using Mojo.js and htmx was more appropriate.

BakeLab mainly focuses on structured forms, bake documentation, and critique workflows rather than highly dynamic real-time interaction. Because of this, a large client-side framework would introduce additional complexity without significantly improving the user experience.

# Choosing Mojo.js

Mojo.js was selected because it supports server-rendered interaction while remaining lightweight and manageable within the assignment scope. Routing, templates, and backend logic remain closely connected, which simplifies debugging and makes the application architecture easier to understand.

For example, when a user submits a bake log, the server can validate the form, store the data in SQLite, and return an updated page without requiring a separate frontend application. Since BakeLab depends heavily on structured data and relational workflows, this architecture fits the project well.

Another important factor was feasibility. The prototype already includes CRUD operations, authentication, database modelling, and interface design. Introducing a more complex frontend stack risked shifting development focus away from the actual critique workflow and toward framework management.

# Using htmx for Interaction

Although BakeLab does not require a fully client-side application, it still benefits from responsive interaction. htmx became useful because it allows sections of a page to update without refreshing the entire interface.

For example, critique comments and validation feedback can update asynchronously after form submission while the rest of the bake detail page remains stable. This creates smoother interaction without introducing frontend state-management complexity.

Another advantage is progressive enhancement. Since the server remains responsible for rendering content, the application can still function even if JavaScript fails. This keeps the system more reliable and easier to maintain during development.

Compared with React, htmx also avoids additional architectural overhead such as component state management, API synchronisation, and client-side routing. For BakeLab’s current scope, these systems would increase implementation complexity without clearly improving the core user workflow.

# Trade-Offs and Reflection

Choosing a simpler architecture also introduced trade-offs. A server-rendered approach may become limiting for highly interactive features such as live collaboration or real-time notifications. However, feasibility analysis showed that implementing these systems would distract from the main goal of supporting structured critique and readable bake documentation.

This process reinforced an important lesson during development: technologies should support user goals rather than being selected because they are popular. Earlier assumptions suggested that adding more advanced frontend systems would make the application feel more modern. However, further planning revealed that increasing architectural complexity was beginning to weaken implementation clarity and project feasibility.

# Conclusion

BakeLab’s technical architecture was shaped by usability, feasibility, and interaction requirements rather than framework trends. Using Mojo.js, SQLite, and htmx allowed the project to support responsive interaction while keeping the application manageable within the semester scope.

More importantly, this stage highlighted how technical decisions directly affect user experience. Simplifying the architecture helped prioritise readability, structured critique, and maintainable interaction instead of unnecessary frontend complexity.
